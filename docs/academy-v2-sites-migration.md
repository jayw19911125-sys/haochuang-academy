## 為何不直接覆蓋舊版

舊版仍有登入、資料庫、測驗、管理後台與歷史進度。直接重寫會同時混入：

- 舊 `deviceId` 身份制度
- 舊版 30 天里程碑
- 舊版 XP 與排行榜
- 已過期 AI 工具矩陣
- 未驗證演算法權重
- 未確認證據的案例資料

因此 V2 先分離介面與制度，再逐項決定哪些資料能遷移。

## ChatGPT Sites 對應方式

| 現有架構 | Sites 建議 |
|---|---|
| Vite／React UI | 優先沿用並做相容性調整 |
| Express／tRPC | 改為 Sites 支援的後端函式或 API |
| MySQL／Drizzle | 評估轉為 D1；正式資料仍按公司治理分流 |
| Manus OAuth | 改成 ChatGPT／工作區身份 |
| deviceId | 移除，改用正式 User ID |
| LocalStorage 學習進度 | 僅保留 UI 偏好；正式考核進度進資料庫 |
| 內部合約模板 | 不搬；只連結正式 Drive 正本 |
| 客戶素材與成品 | 不搬；保留 Drive／NAS |
| 任務與專案狀態 | 不搬；Monday 為正本 |
| SOP 與教學 | Notion 為正本，Site 作互動入口與摘要 |

## 正式資料邊界

### Site 可保存

- 使用者職務
- 能力模組狀態
- 1／3／6 個月考核狀態
- 主管回饋
- 工作證據連結
- 教材版本與閱讀狀態

### Site 不應保存

- 客戶帳密與 API Key
- 正式合約與發票
- 銀行與財務明細
- 原始素材與大型成品
- 未脫敏客戶機密
- Monday 任務的另一份複製資料
